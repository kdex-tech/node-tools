# Runtime benchmark: `optimize` / `generate` / `bundle_cjs`

Compares JavaScript runtimes executing the three packaging commands, for
both **compatibility** (does it run correctly) and **speed**.

## Environment

- Host: Linux x64
- node: v22.22.2
- deno: deno 2.9.1 (stable, release, x86_64-unknown-linux-gnu)
- bun: 1.3.11
- Fixture: html-parse-stringify, i18next, react, react-dom, react-i18next, void-elements
- Iterations: 8 (warmup 2); times are wall-clock medians including runtime startup.

## Compatibility

| command | node | deno | bun |
|---------|-----|-----|-----|
| optimize | ✅ | ✅ | ✅ |
| generate | ✅ | ✅ | ✅ |
| bundle_cjs | ✅ | ✅ | ✅ |

## Speed (median ms, lower is faster; ratio vs node)

| command | node | deno | bun |
|---------|-----|-----|-----|
| optimize | 1607 (1.00x) | 1699 (1.06x) | 1646 (1.02x) |
| generate | 241 (1.00x) | 263 (1.09x) | 206 (0.86x) |
| bundle_cjs | 282 (1.00x) | 260 (0.92x) | 217 (0.77x) |

## Findings

- **optimize**: runtime-bound — all runtimes within 6% (~1607ms). Dominated by the native esbuild binary, so the JS runtime barely matters here.
- **generate**: fastest is **bun** at 206ms (14% faster than node). JS-bound, so runtime speed shows.
- **bundle_cjs**: fastest is **bun** at 217ms (23% faster than node). JS-bound, so runtime speed shows.

- **Compatibility**: every runtime ran every command correctly, with identical output (same files rewritten, same import map). esbuild's native binary and cjs-module-lexer work under all three; deno needs `-A --node-modules-dir=manual`.
- **Recommendation**: on the JS-bound commands, **bun** is the fastest and a drop-in replacement. The big win — `optimize` — is esbuild-bound and won't move by swapping runtimes; to speed it up, batch the per-file esbuild calls instead.

_Generated by `bench/runtime/run.js`. Absolute numbers depend on the host;
the ratios between runtimes on the same machine are the meaningful signal._
